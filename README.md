# Intelligent Agents — Cooperative Cognitive Platform Simulation

This project is a PyQt6/pyactr reference environment for two-agent cooperation in a continuous, gravity-driven platform world. It replaces the former matrix demo while retaining the unified simulation, stepper, history, buffer inspection, graph analysis, multi-run, and optional pyactr performance-override architecture.

## Core game model

Every run contains exactly two player entities. They can be either two ACT-R agents or one ACT-R agent plus one human-controlled agent. At the beginning of every procedurally generated level, the two entities are randomly assigned to:

- **Circle**: moves with A/D, jumps with W, and accelerates downward with S. Every accepted jump samples a bounded vertical and horizontal impulse, so jump height and distance vary while remaining controllable.
- **Rectangle**: moves with A/D. W makes it taller and narrower; S makes it flatter and wider. Its initial width and maximum height are randomized per level. Giant-lift scenarios permit a visual maximum height of more than 40 world units, although nearby ceilings and ledges may limit the locally usable transformation.

The circle can stand on the rectangle. Transforming the rectangle while the circle is on top lifts the circle, creating the required cooperative stack mechanic.

## Physics and performance

The world is continuous rather than cell based. A fixed-step solver runs at 120 Hz and handles:

- gravity and terminal velocity;
- axis-aligned solid-platform collision;
- circle/rectangle support and carrying;
- smooth rectangle transformation with collision rejection;
- collectible overlap;
- deterministic model-time advancement for ACT-R-only runs;
- wall-clock advancement for human sessions.

A general-purpose physics package is intentionally not required. With only two dynamic bodies and a small number of static rectangles, the specialized solver avoids broad-phase/indexing overhead and keeps batch runs deterministic. Existing dependencies therefore remain sufficient.

## Procedural level guarantees

Levels are randomized inside a validated scenario grammar rather than generated by unconstrained scatter. The grammar rotates deterministically through eight topology families: paired-left, paired-right, opposed upper starts, separate chambers, a central split, a giant-lift layout, and two vertical-reunion variants. Agent order can be mirrored, one agent can begin on a different tier, and either random avatar assignment remains supported. Every generated level contains:

1. solo-accessible diamonds;
2. a low route that rewards flattening the rectangle;
3. upper and lower diamonds above the circle's solo jump envelope but within the combined tall-rectangle-plus-jump envelope;
4. a priority diamond before an irreversible descent;
5. a lower basin that cannot be climbed back to the upper section.

Collecting all diamonds starts the next generated level automatically. Press **R** in a human session to restart the current seed after missing a priority diamond or becoming trapped.

## ACT-R visual translation

The backend never exposes a movement matrix. `Middleman.get_agent_stimulus()` converts continuous geometry into compact pyactr visual data:

- positions are quantized to quarter-world-unit coordinates;
- each agent emits a stable self-relative identity stimulus (`A` for self, `B` for the partner), a shape stimulus (`C` or `R`), posture information for the rectangle, and visible body-bound stimuli;
- every diamond emits `D` with a semantic stimulus identifier containing its target identity, role, and required order;
- each large platform emits `P` at its left edge, center, and right edge, with its platform kind encoded in the visual stimulus identifier;
- `ExampleAdapter` receives only the sanitized pyactr frame through `pyactr_extension.visual_stimuli()`. The richer `visual_metadata` inspection channel is not accessible to the cognitive adapter.

The inspector's character raster is only a local visualization of these continuous stimuli; it is not the simulation state representation.

## Cognitive Example agent

`agents/Example.py` and `agents/ExampleAdapter.py` provide the executable reference controller for this world. The implementation follows a strict separation of concerns:

- **Productions own cognitive control and motor choice.** Identification, assessment, strategy retrieval, conflict resolution, locomotion, transformation, jump timing, waiting, replanning, and emergency restart are explicit named productions with a utility hierarchy.
- **The adapter is a computation bridge, not a hidden policy.** It reads only the paired model's chunk protocol and the pyactr visual frame exposed by `AgentConstruct`. It has no references to the environment, physical entities, the partner construct, `middleman`, or `visual_metadata`, and it never sends motor commands.
- **Strategy knowledge is retrieved from declarative memory.** Before acting, the model retrieves a `strategy_schema` chunk such as `direct_circle`, `cooperative_rectangle`, `cooperative_circle`, `descend_after_priority`, or `recover_restart`.

The cognitive cycle is:

1. a production writes an `identify` or `assess` request into the `adapter_protocol` buffer;
2. the adapter reconstructs the visible scene exclusively from pyactr stimulus identifiers, symbols, and quantized positions;
3. arithmetic and logical extension functions update self, partner, world, target, candidate, and affordance chunks;
4. the adapter marks the protocol request as complete;
5. productions confirm the result, retrieve a `strategy_schema`, resolve action utilities, and issue motor commands through pyactr's manual buffer;
6. the production system returns to assessment after the manual module becomes free.

The internal adapter bridge is deliberately omitted from the State Graph because it is not an official pyactr module. The graph shows only ACT-R productions and accesses such as Goal, Retrieval, Declarative Memory, Visual, and Manual.

### Mental model chunks

The live buffers contain:

- `self_model`: identity, assigned shape, current and previous visible position, derived displacement, support, transform, region, motion, confidence, and perception revision;
- `other_model`: the same observable state plus inferred intention, inferred target, readiness, spatial relation, route-blocking relation, reciprocal commitment, and confidence;
- `world_model`: remaining upper/lower targets, priority state, safe-descent state, fall risk, room/tier separation, reunion requirement, cooperation availability, and perception revision;
- `candidate_model`: the currently preferred target, reachability class, actor assignment, strategy, score, distance, staging point, reservation, and rationale;
- `coordination_model`: joint target, current social phase, complementary roles, commitments, blocker identity, yield direction, partner direction, separation, progress, timeout, and retrieved association;
- `affordance_model`: simultaneously available actions and timing predicates.

Declarative memory additionally contains stable `environment_rule`, `strategy_schema`, `coordination_schema`, `social_episode`, `metric_parameter`, and `target_policy` chunks, plus dynamically refreshed `platform_model`, `target_model`, `self_model`, `other_model`, `world_model`, and `coordination_model` chunks. This makes mechanics, numerical thresholds, target policies, theory of mind, and action rationale visible in the GUI inspectors and graphs.

### Cooperation and anticipation

For stack targets, both agents bind to the same joint target through complementary role and commitment chunks. The rectangle stages beside the ledge while flat. The circle follows, mounts it, and centers itself. The rectangle raises only while the target remains outside the rider's visual jump envelope; the circle waits until the target becomes reachable and then issues a timed directional jump. This target-relative criterion remains valid even when the rectangle's theoretical maximum is much larger than the local ceiling clearance. A failed mount is recoverable because the rectangle deliberately flattens again.

Route conflicts are also cognitive states. If a supporting body lies between the collector and its target, the visual frame updates reciprocal blocking slots in `other_model`, selects the `blocking_requires_yield` association from `coordination_schema`, and productions `P34`/`P35` move the blocker behind the collector. If agents begin in different rooms or tiers, `separation_precedes_cooperation` activates the `reunite_team` production family before target execution.

The partner model is inferred from the partner's visual identity, shape, body bounds, posture, position history, support geometry, target fit, and cooperation geometry. Neither adapter can access the other agent's buffers or construct.

### Arithmetic and logical extension

`simulation/integrations/pyactr_extension.py` provides the adapter-safe operations used during assessment: numeric conversion, addition, subtraction, multiplication, division, minimum/maximum, absolute value, square, tolerant comparison, symbolic Boolean logic, clamping, Euclidean distance, and directional classification. Symbolic values such as `yes`, `no`, `true`, `false`, `none`, and `0` are interpreted explicitly rather than through Python's truthiness rules.

## Human controls

| Key | Circle | Rectangle |
|---|---|---|
| A / D | Move left / right | Move left / right |
| W | Randomized bounded jump | Taller and narrower |
| S | Fast-fall / brake on ground | Flatter and wider |
| R | Restart current level | Restart current level |

Arrow keys mirror WASD.

## Running

```bash
python -m pip install -r requirements.txt
python main.py
```

The GUI starts maximized. Configure exactly two total player slots before starting. For a human session, enable the human player and leave exactly one ACT-R instance active.

## Project structure

- `simulation/world/entities.py`: continuous entities and physics bodies.
- `simulation/world/level_builder.py`: randomized, validated level construction.
- `simulation/world/environment.py`: fixed-step physics, controls, collection, progression, and snapshots.
- `simulation/runtime/middleman.py`: ACT-R motor and visual translation.
- `agents/Example.py`: production system, chunk schemas, declarative mechanics, and strategy knowledge.
- `agents/ExampleAdapter.py`: visual-frame parsing, arithmetic/logical derivation, protocol completion, theory-of-mind updates, and chunk maintenance.
- `simulation/integrations/pyactr_extension.py`: pyactr compatibility fixes and deterministic arithmetic/logical bridge.
- `gui/environment_canvas.py`: cached continuous-world renderer.
- `gui/environment_view.py`: environment metadata and control legend.
- `pyactr_overrides/`: reversible experimental pyactr optimizations, activated only by the GUI toggle.

## Validation

Run the regression suite with:

```bash
python -m unittest discover -s tests -v
```

The suite checks hundreds of generated levels across all eight scenario families, large transformation envelopes, physics invariants, restart behavior, extension arithmetic and logic, production/chunk architecture, adapter isolation, semantic visual stimuli, protocol-free State Graph exports, saved-LOS compatibility, real Manual commands, route-yield behavior reconstructed from the supplied history, and timed raise/jump cooperation.

## Multi Simulation Run

Headless scenarios use two ACT-R controllers. If an interactive configuration contains one ACT-R agent and one human player, the batch conversion replaces the human slot with a second instance of the configured ACT-R model. Physics advances according to ACT-R model time, and continuous environment snapshots are included in exported histories.
